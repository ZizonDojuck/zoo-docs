## 목적 (왜)

주가는 종목 뉴스만으로 안 움직임. 금리·법령·환율·지정학·섹터 등 거시·경제가 시장 방향을 정함(미국 가치주 4기둥 ③미래성장성·④타이밍 / 한국단타 급락장 회피의 근거). 그래서 뉴스를 **실시간으로 읽고 버리는 게 아니라 데이터화해서 저장 → 필요할 때 호출**하는 데이터센터를 만든다. 이 1차 작업은 **수집 → 정규화 → 저장**까지. 점수화(E5·sentiment)·매매판단 연결은 후속.

설계 원칙(확정됨):
- **경제뉴스 = 메인축**(거시·정책·금리·환율·법령), 섹터·종목 = 보조축.
- **다소스 광범위 수집**(외신 포함) — 매체별 편향 상쇄·다각도. 한국 정보 은폐 보완 위해 외신 병행.
- 3계층: L1_macro(거시) / L2_sector(섹터) / L3_stock(종목).
- 다소스 중복은 **이벤트 클러스터링**으로 묶음(같은 사건 = 1 cluster, 보도 매체 수 = 주목도).
- L1은 넓게, L2·L3는 보유 우주(반도체·빅테크·AI, US_PORTFOLIO)로 좁게.

## 소스 (실측 완료 — 추정 아님)

채팅 단계에서 실제 호출로 응답 구조 확인 완료. 아래는 실측된 사실:

### S1. FMP 뉴스 (미국·글로벌 메인) — ✅ 동작확인
- 엔드포인트: `general-news`(거시·일반), `search-stock-news`(symbols 지정, 종목), `press-releases`(공시·보도자료), `forex-news`(환율). from_date/to_date/limit/page 지원(과거 소급 가능).
- 응답 필드: `symbol`(거시=null, 종목=티커), `publishedDate`("2026-06-27 07:30:00"), `publisher`, `title`, `image`, `site`, `text`(요약), `url`.
- **symbol null 여부로 L1/L3 1차 자동분류 가능.**
- .env에 FMP_API_KEY 이미 있음(가치주 E4에서 사용 중).

### S2. 네이버 뉴스 검색 (한국 메인) — ✅ 동작확인
- PlayMCP 경유 `NaverSearch-search_news`(query, display, sort=date/sim, start). 토스 화이트리스트와 무관한 별도 경로.
- 응답 필드: `total`(검색량=주목도 지표), `items[]` 각각 `title`(★`<b>`태그 포함), `originallink`(원문), `link`(네이버), `description`(요약, ★`<b>`태그), `pubDate`("Sat, 27 Jun 2026 14:40:00 +0900" RFC822 형식).
- **★주의: title·description에 `<b></b>` 태그가 박혀서 옴 → 저장 전 태그 제거 필수.**
- **★pubDate가 RFC822 형식 → ISO8601 변환 필요.**
- **★total은 그 검색어의 사회적 관심량 → attention 필드로 활용(E5 입력값).**
- 단, PlayMCP 호출 한도·안정성은 실측 대상(아래 R5).

### S3. yfinance 종목 뉴스 (교차검증 보조) — ✅ 동작확인 / 단 약점 있음
- PlayMCP `UsStockInfo-get_finance_news(ticker)`.
- 응답: Title, Summary, URL. **★발행일·매체명 없음 → 시계열 정렬·신뢰도 측정 불가. 메인 적재 부적합, 교차검증 보조로만.**

## 확정 저장 스키마 (이 형태로 정규화해서 저장)

각 소스 응답을 아래 **공통 스키마로 변환(정규화 어댑터)** 후 저장. 소스마다 필드명이 다르므로 어댑터 필수.

```json
{
  "id": "sha1(title+published) 앞 12자 — 중복적재 방지 키",
  "layer": "L1_macro | L2_sector | L3_stock",
  "title": "HTML태그 제거된 제목",
  "summary": "HTML태그 제거된 요약",
  "published": "ISO8601 (소스별 날짜형식 통일)",
  "source": "fmp | naver | yfinance",
  "publisher": "Market Watch / 연합뉴스 등 (없으면 도메인 추출)",
  "url": "원문 링크",
  "symbols": ["NVDA", "TSM"],
  "sector_tags": ["반도체", "AI"],
  "region": "US | KR | GLOBAL",
  "attention": 142037,
  "cluster_id": "같은 이벤트 묶음 키",
  "source_count": 1,
  "sentiment": null,
  "score": null
}
```

필드 규칙:
- `id`: 중복 적재 방지. 같은 id면 재저장 안 함(스킵).
- `layer`: FMP symbol null→L1 후보 / symbol 있음→L3. 섹터 키워드 매칭 시 L2. 분류 규칙은 R3.
- `title`/`summary`: **HTML 태그·엔티티 제거(sanitize) 후 저장.** 네이버 `<b>` 제거 필수.
- `published`: 전 소스 ISO8601로 통일. yfinance는 발행일 없음→수집시각으로 대체하되 `published_missing:true` 플래그 남길 것(추정 금지 원칙).
- `symbols`/`sector_tags`: 태그 참조. L1 뉴스도 관련 종목·섹터 있으면 태그(예: "반도체 관세" → sector_tags:[반도체], symbols:[TSM,MU,NVDA]). 무리한 추정 매칭 금지 — 명시적 언급만.
- `region`: 소스·내용 기반(FMP general=US/GLOBAL, 네이버=KR).
- `attention`: 네이버 total 있을 때만. 없으면 null.
- `cluster_id`/`source_count`: R2 클러스터링.
- `sentiment`/`score`: **이번엔 null로 예약.** 채우지 말 것(후속 E5 작업).

## 작업 항목

### R1. 디렉터리 구조 + 저장 형태
- `news/` 디렉터리 생성(메모리 폴더 하위).
- 저장 형태 권장: **날짜별 JSONL**(`news/2026-06-27.jsonl`, 한 줄 1 뉴스) — append 효율·중복체크 용이. 또는 종목별 분리. **ClaudeCode 판단으로 효율적인 쪽 선택하되 이유를 보고에 1줄.**
- 용량 의식: 원문 전체 저장 금지. title+summary+메타만(구조화 요약). 연 70MB 이하 목표.
- **★나중에 깃헙 백업 예정** → news/ 데이터에 키·토큰 절대 섞이지 않게(.env 분리 유지). news 데이터 파일은 .gitignore 대상 아님(코드와 함께 커밋될 자산).

### R2. 정규화 어댑터 + 이벤트 클러스터링
- 소스별 어댑터: `normalize_fmp()`, `normalize_naver()`, `normalize_yf()` → 공통 스키마 반환.
- 클러스터링: 같은 사건을 보도한 다소스 뉴스를 1 cluster로. 1차 구현은 **제목 유사도(간단한 토큰 자카드/정규화 후 비교) + 같은 날짜** 기준이면 충분. 정교한 NLP 불필요(과설계 금지). 같은 cluster면 source_count 증가, cluster_id 공유.
- **수집 실패 시 추정 금지**: 어떤 소스가 실패하면 그 소스만 건너뛰고 "수집 실패" 기록. 빈 데이터를 추정으로 채우지 말 것. 전 소스 실패 시 텔레그램 "뉴스 수집 실패" 알림(메모리 원칙).

### R3. 3계층 자동분류 규칙
- L1_macro: FMP symbol=null AND 거시 키워드(금리/Fed/FOMC/CPI/관세/tariff/환율/규제 등) / 네이버 경제 카테고리.
- L3_stock: 특정 종목(symbol 또는 본문 명시) 직결.
- L2_sector: 섹터 키워드(반도체/semiconductor/AI/빅테크 등) 매칭, 특정 종목 아님.
- 중복 가능(한 뉴스가 L1이면서 종목 태그 보유 가능) → layer는 1차 성격, symbols/sector_tags로 교차연결.
- 분류 키워드는 상수로 분리(잠정·조정가능).

### R4. KIS·토스 뉴스 엔드포인트 실측 (❓→✅/⬜ 확정)
- KIS OpenAPI 공식 문서/샘플에 국내뉴스·종목뉴스 시세성 뉴스 엔드포인트가 있는지 실측. 있으면 소스 추가(S4), 없으면 ⬜ 확정하고 보고.
- 토스 OpenAPI에 뉴스 엔드포인트 있는지 실측. 메모리상 토스는 시세·보유 중심이라 없을 가능성 높음 → 없으면 ⬜ 확정. 수동 캡처 폴백.
- **실측 결과를 ✅(있음+동작)/⬜(없음)/🔶(있으나 제약)로 보고.** 추정으로 "있다/없다" 단정 금지.

### R5. 1회 수집 테스트 (실측)
- 위 어댑터로 FMP general-news + search-stock-news(US_PORTFOLIO 일부) + 네이버(경제 키워드) 실제 1회 수집 → 정규화 → news/에 저장까지 돌려볼 것.
- PlayMCP 네이버 호출 한도/안정성 실측(연속 호출 시 막히는지).
- 저장된 레코드 샘플 2~3건 + 클러스터링 동작(같은 이벤트 묶였는지) 확인.
