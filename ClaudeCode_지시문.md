> **기준**: 전문적·엄격. 오타·누락·보안사고 금지. 추정을 사실로 쓰지 말 것.

## A. 이번 작업의 핵심 (우선순위 순)

### A-1. 요구사항 추적표 R1~R26 실측 갱신 (최우선)
`CLAUDE.md` 0.2절의 추적표는 현재 거의 전부 ❓ 상태다.
**코드를 실제로 열람**하여 각 행을:
- 상태: ✅완료 / 🔶부분 / ⬜미착수 로 갱신
- 검증근거: `파일명:함수명` 또는 실호출 테스트 결과 기입

→ 채팅 Claude는 코드를 볼 수 없다. 네가 까서 보고해야 검증이 가능하다.

### A-2. 자가진단 대시보드 신설 (R26) — 이번 작업의 주 산출물
`CLAUDE.md` 7장 명세대로 2종 산출:

1. **`self_check.html`** — 시각적 점검판
   - R1~R26 구현여부 (색상: GREEN=✅ / YELLOW=🔶 / GRAY=⬜ / RED=❓·FAIL)
   - 데이터 연결 **실호출 테스트** 결과 표:
     네이버 / pykrx / yfinance / OpenDART / 토스OpenAPI → 각각 PASS/FAIL + 응답시간(ms) + 마지막 성공시각
   - 최근 에러로그 N건
   - 보안 검증 5단계 통과 여부 (체크리스트)
   - 종합 상태등: GREEN / YELLOW / RED
   - **주의**: 정적 HTML로 생성. MCP/외부 API를 fetch로 직접 호출하는 코드 금지.
     테스트 결과는 파이썬에서 미리 수집해 HTML에 주입할 것.

2. **`self_check.md`** — 채팅 Claude가 읽고 의도 검증할 텍스트 리포트
   - 각 R항목: `구현위치(파일:함수) | 상태 | 실측근거`
   - 미구현/부분구현 목록을 별도 섹션으로 (채팅 Claude 검토 대상)

3. 실행 스크립트(예: `self_check.py`)로 위 2종을 생성.
   - 데이터 실호출 테스트는 **조회 전용**.
   - **토스 주문 호출은 어떤 경우에도 포함 금지.**

### A-3. 토스 Open API 실접근 검증 (R21)
순서 엄수: ①OAuth2.0 인증 토큰 발급 → ②시세 조회 → ③잔고 조회.
**주문 호출(매수/매도)은 절대 실행하지 말 것 (R22).**
- 주문 관련 코드가 있다면 `DRY_RUN=True` / 플래그 off 기본값 확인 및 보고.
- 검증 결과(성공/실패, 토큰 발급 여부, 수급 데이터 부재 확인)를 self_check에 반영.

## B. 검증·진단 시 지켜야 할 원칙

1. **추정 금지**: 확인 안 된 것은 self_check.md에 "(미확인)"으로 명시. 동작한다고 단정 금지.
2. **부분동작 ≠ 완성**: 일부만 되면 🔶로 표기. ✅는 완전 동작 + 근거 있을 때만.
3. **수집 실패 시 GO 금지(R7)**: 데이터 수집 실패 경로가 코드에 구현됐는지 확인.
   실패 시 매매판단 보류 + 텔레그램 "수집 실패" 알림이 실제 작동하는지 테스트.
4. **how 담당**: 너는 "코드가 어떻게 동작하는가"를 진단한다.
   "원래 의도와 맞나/빠진 요구 없나"(what·why)는 채팅 Claude가 self_check.md를 받아 판단한다.

## C. 보안 검증 5단계 (커밋·업로드 전 매번)

1. `.gitignore`에 `.env`, `*.key`, `secrets/` 등록 확인
2. 키 노출 정규식 스캔 (토큰·API키·계좌번호 패턴) — 결과 보고
3. `git status`로 `.env` 미staged 확인
4. GitHub Actions 워크플로에 평문 키 없는지, Secrets 주입인지 확인
5. 노출 발견 시 **즉시 키 재발급** + 히스토리 정리 + 이주혁에게 보고

→ 위 5단계 통과 여부를 self_check.html 보안 섹션에 반영.

## D. 데이터 소스 점검 체크 (실호출)

| 소스 | 테스트 내용 | 기대 결과 |
|------|-------------|-----------|
| 네이버 API | 개별종목 수급·현재가·일봉 1건 조회 | PASS + 데이터 반환 |
